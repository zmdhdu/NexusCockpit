## 适用场景

- 小白开发者初次阅读 NexusCockpit 项目代码时，需要逐行注释帮助理解。
- 为复杂算法、异步逻辑、设计模式添加"大白话"注释。
- 将英文技术术语翻译为中文并附上通俗解释。
- 为配置文件（.env、yaml、json）的每一项添加说明。
- **新增: 生成模块学习卡片** — 为指定模块生成"是什么/为什么/怎么做"三段式学习卡片。
- **新增: 绘制数据流图** — 用 ASCII 字符画绘制请求从入口到出口的完整数据流。
- **新增: 概念速查** — 对项目中出现的技术概念给出通俗类比解释。

## 非适用场景

- 不用于生成正式 API 文档（请使用 `code-doc` 技能）。
- 不用于检测代码质量（请使用 `code-review` 技能）。
- 不用于同步文档与代码一致性（请使用 `doc-sync` 技能）。

## 模式一: 代码注释 (原有能力)

### 1. Python 代码注释

**函数注释 — 通俗版**:
```python
def search_memory(query: str, user_id: str, top_k: int = 3) -> list:
    """
    在记忆库里搜索和用户问题最相关的历史记忆。

    打个比方：这就像在你的笔记本里搜索和当前问题相关的页码。
    系统会把用户的问题转成向量（一串数字），然后在 Milvus 数据库里
    找出最相似的几条记忆。

    参数:
        query: 用户的问题文本，比如 "我昨天设的空调温度是多少"
        user_id: 哪个用户在问，每个用户的记忆是隔离的
        top_k: 最多返回几条记忆，默认3条

    返回:
        一个列表，里面是找到的记忆，每条记忆包含:
        - text: 记忆的内容
        - score: 相似度分数（越高越相关）
    """
```

**行内注释 — 解释式**:
```python
# === 1. 把用户的文字问题转成向量（一串2560个数字）===
# 就像把一句话翻译成机器能理解的"密码"
embedding = await self._embed_text(query)

# === 2. 在 Milvus 向量数据库里搜索相似的记忆 ===
# Milvus 会计算向量的距离，距离越近 = 内容越相关
results = self.collection.search(
    data=[embedding],          # 搜索用的向量
    anns_field="vector",       # 在哪个字段搜索
    limit=top_k,               # 最多返回几条
    filter=f'user_id == "{user_id}"',  # 只搜这个用户的记忆
)
```

  })
}

## 模式二: 模块学习卡片 (新增)

当用户说"学习某个模块"或"解释某个文件"时，生成以下格式的学习卡片:

### 学习卡片模板

```markdown
## 📦 模块学习卡片: [模块名]

### 🎯 是什么
[用 1-2 句大白话描述这个模块的作用]

### 🤔 为什么需要它
[解释如果没有这个模块会怎样，它解决了什么问题]

### 🔧 怎么工作的
[用编号步骤描述核心流程，每步配一句大白话]

### 📁 关键文件
| 文件 | 作用 | 行数重点 |
|------|------|----------|
| xxx  | xxx  | 第 N-M 行 |

### 🔗 和谁交互
- 上游（谁调用它）: xxx
- 下游（它调用谁）: xxx

### 💡 生活类比
[用一个生活中的例子类比这个模块的作用]

### ⚠️ 注意事项
[新手容易踩坑的地方]
```

### 示例: SupervisorGraph 学习卡片

```markdown
## 📦 模块学习卡片: SupervisorGraph (Agent编排核心)

### 🎯 是什么
这是整个 AI 系统的"大脑调度中心"，负责决定用户的请求交给哪些专家处理、
怎么汇总结果、怎么生成最终回复。

### 🤔 为什么需要它
如果只有一个 AI 处理所有问题，它可能什么都懂一点但什么都不精。
就像医院的分诊台 — 你说肚子痛，它把你送到消化科；你说头痛，送到神经科。
Supervisor 就是这个分诊台，5 个专家就是 5 个科室。

### 🔧 怎么工作的
1. 接收用户输入 → 去记忆库里找有没有相关历史
2. 意图路由 → 判断该交给哪些专家（可能多个同时激活）
3. 专家并行 → 5 个专家同时工作（车控+导航+闲聊...）
4. 汇总回复 → Responder 把所有专家的结果合并成一条回复
5. 质量审查 → Reviewer 检查回复质量，存入记忆
6. 流式返回 → 通过 SSE 逐字推送给前端

### 📁 关键文件
| 文件 | 作用 | 行数重点 |
|------|------|----------|
| supervisor_graph.py | 编排核心 | 第 59-80 行: 类定义和初始化 |
| experts/base.py | 专家基类 | 第 1-50 行: 接口定义 |
| responder.py | 回复汇总 | 第 30-60 行: LLM 流式生成 |
| reviewer.py | 质量审查 | 第 20-40 行: 审查规则 |

### 🔗 和谁交互
- 上游: API 路由 (chat.py) 调用 `agent_graph.invoke()` 或 `stream()`
- 下游: MemoryManager (记忆)、IntentRouter (意图)、SkillRegistry (技能)

### 💡 生活类比
想象一个公司的客服中心:
- Supervisor = 前台接线员（判断转给谁）
- Experts = 5 个部门（维修/导航/生活/医疗/闲聊）
- Responder = 部门经理汇总意见写回复
- Reviewer = 质检员检查回复质量

### ⚠️ 注意事项
- 专家是**并行**执行的，不是排队
- state 是 TypedDict，运行时是字典，用 `state["key"]` 而不是 `state.key`
- v2.1 增加了 MainAgent 确认层，某些场景需要二次确认才放行
```

## 模式三: 数据流图 (新增)

当用户说"画一下数据流"或"请求怎么走"时，用 ASCII 字符画绘制:

### 数据流图模板

```markdown
## 🔄 数据流图: [场景名]

### 请求路径
```
[入口] 用户操作
  │
  ▼
[层1] 组件名          ← 做了什么
  │   ├─ 子步骤 1
  │   └─ 子步骤 2
  │
  ▼
[层2] 组件名          ← 做了什么
  │
  ▼
[出口] 返回结果
```

### 关键数据结构
```json
{
  "field": "说明"
}
```

### 异常路径
```
[入口] → [层1] → ❌ 失败 → [异常处理] → 返回错误
```
```

### 示例: 用户对话数据流

```
用户在聊天框输入 "把空调调到24度"
  │
  ▼
[前端] chat-window.tsx
  │   ├─ addMessage() 添加用户消息到列表
  │   └─ streamMessage() 发起 SSE 请求
  │       └─ 附加 JWT Token + X-Cockpit-Id 头
  │
  ▼
[Go网关] router.go
  │   ├─ AuthMiddleware() 验证 JWT
  │   ├─ RateLimitMiddleware() 限流检查
  │   └─ proxyToPython() 转发到 Python
  │
  ▼
[Python] chat.py → supervisor_graph.py
  │   ├─ SemanticCache 查语义缓存 (Redis KNN)
  │   │   ├─ 命中 → 直接返回缓存结果
  │   │   └─ 未命中 ↓
  │   ├─ Supervisor 节点
  │   │   ├─ MemoryManager.recall() 召回记忆
  │   │   ├─ IntentRouter.route() 意图路由 → vehicle_expert
  │   │   └─ 设置 active_experts = ["vehicle"]
  │   ├─ VehicleExpert 节点 (并行)
  │   │   ├─ SkillRegistry 调用 climate 技能
  │   │   └─ VehicleAdapter.send_command("set_temp", 24)
  │   ├─ Responder 节点
  │   │   └─ LLM 流式生成回复 (SSE chunks)
  │   └─ Reviewer 节点
  │       └─ MemoryManager.store() 存入记忆
  │
  ▼
[前端] 逐块接收 SSE 事件
  │   ├─ updateMessage() 更新 AI 消息内容
  └─ TTS 语音播报 (Web Speech API)
```

## 模式四: 概念速查 (新增)

当用户问到项目中某个技术概念时，用以下格式解释:

### 概念速查模板

```markdown
## 💡 概念速查: [术语]

**一句话解释**: [用最简单的话说清楚]

**生活类比**: [用日常生活中的例子打比方]

**在项目中哪里用了**: [文件路径 + 简短说明]

**为什么用这个而不是其他方案**: [对比说明]

**深入学习链接**: [项目内文档路径]
```

### 示例

```markdown
## 💡 概念速查: GraphRAG

**一句话解释**: 同时用"语义搜索"+"关系图谱"+"关键词匹配"三种方式找知识，然后融合排序。

**生活类比**: 你要找一本书:
- 向量搜索 = 按内容相似度找（"有没有讲类似主题的"）
- 图谱搜索 = 按关系找（"这个作者还写了什么"）
- BM25 = 按关键词找（"标题里有'Python'的"）
- RRF融合 = 综合三种方法的排名，取最优
- Rerank = 最后请专家再精选一遍

**在项目中哪里用了**: `nexus/rag/retriever.py` 的 `GraphRAGRetriever` 类

**为什么用这个而不是其他方案**: 单一搜索方式各有盲区:
- 纯向量: 可能找到语义相似但不相关的
- 纯关键词: 找不到同义词
- 纯图谱: 需要预先建立关系
- 三路融合 + Rerank = 取长补短，准确率最高

**深入学习链接**: `docs/architecture/L2-data.md`
```

## 注释原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **大白话** | 用生活类比解释技术概念 | "向量就像把句子翻译成机器能理解的密码" |
| **分层注释** | 先模块级概述，再函数级说明，最后行内细节 | `=== 1. xxx ===` 标记步骤 |
| **标注术语** | 英文术语首次出现时附中文翻译 | `Embedding（嵌入/向量化）` |
| **解释 Why** | 注释重点解释"为什么这样做"而非"做了什么" | "用异步是为了不阻塞主线程" |
| **适度注释** | 不是每行都注释，只注释非显而易见的逻辑 | 简单赋值不需要注释 |
| **标量关系** | 标注数据从哪来、到哪去 | `# ← 从 chat.py 传入` |
| **标注版本** | v2.0/v2.1 新增的逻辑要标注 | `# v2.1 新增: 多座舱隔离` |

## 执行步骤

### 当用户要求"注释代码"时:
1. **通读模块**：先阅读整个文件，理解模块整体职责。
2. **添加模块头注释**：在文件开头用 3-5 句话说明这个模块做什么。
3. **标注关键函数**：为每个类和公开函数添加通俗版 docstring。
4. **补充行内注释**：用 `# === N. xxx ===` 格式标注关键步骤。
5. **翻译术语**：将英文术语翻译并附上通俗解释。
6. **检查可读性**：确保一个编程新手能看懂注释后的代码。

### 当用户要求"学习模块"时:
1. **定位模块**：确定用户要学习的模块名或文件路径。
2. **通读代码**：阅读该模块的所有相关文件。
3. **生成学习卡片**：按照"模块学习卡片"模板输出。
4. **绘制数据流**：如果模块涉及数据流转，画出 ASCII 数据流图。
5. **解释概念**：对模块中的技术概念给出通俗解释。
6. **指引下一步**：建议用户接下来看哪个关联模块。

### 当用户要求"画数据流"时:
1. **确定场景**：明确用户要看哪个场景的数据流（如对话、车控、登录等）。
2. **追踪路径**：从前端入口开始，逐层追踪到后端再返回。
3. **绘制图表**：用 ASCII 字符画绘制完整数据流图。
4. **标注关键数据**：在图中标注关键的数据结构和字段。
5. **标注异常路径**：画出失败场景的处理路径。

### 当用户问到"某个概念是什么"时:
1. **定位概念**：确定用户问的是哪个技术概念。
2. **通俗解释**：用一句话 + 生活类比解释。
3. **关联项目**：指出在项目中哪里用了。
4. **对比方案**：说明为什么选这个方案。
5. **指引深入**：给出项目内的学习链接。
