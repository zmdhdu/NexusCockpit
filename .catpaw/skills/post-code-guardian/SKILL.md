---

## 权威入口

- `.catpaw/skills/post-code-guardian/SKILL.md`

## 角色定位

post-code-guardian 是一个 **编排型 SubAgent**，本身不直接审查代码或编写文档，而是按固定流水线依次调度三个子技能，并汇总它们的输出。相当于 CI/CD 流水线中的"编排器"角色。

```
代码修改完成
     │
     ▼
┌─────────────────────────────────────────────┐
│         post-code-guardian (编排器)          │
│                                             │
│  Step 1: 调度 code-review                   │
│          → 检测 bug / 安全漏洞 / 性能问题     │
│          → 如有 Critical/High 问题 → 立即修复 │
│                                             │
│  Step 2: 调度 code-doc                      │
│          → 为新增/修改的函数补充 docstring    │
│          → 为复杂逻辑添加行内注释             │
│                                             │
│  Step 3: 调度 doc-sync                      │
│          → 检查 .md 文档与代码是否一致        │
│          → 自动更新过期文档                  │
│                                             │
│  Step 4: 汇总输出守护报告                    │
└─────────────────────────────────────────────┘
     │
     ▼
  代码质量闭环完成，可安全提交
```

## 适用场景

- **每次代码修改完成后自动触发**（作为收尾必执行步骤）
- Agent 修改了 `.py` / `.ts` / `.tsx` 等源码后的质量保障流水线
- 用户显式要求"检查代码"或"质量保障"时
- `git commit` 前的最终质量门禁

## 非适用场景

- 不用于替代单元测试或集成测试
- 不用于架构设计或技术选型
- 不用于前端 UI 设计（使用 `fronted-design` 技能）
- 不用于快速开发脚手架（使用 `rapid-dev` 技能）

## 前置条件

在调用本技能前，需确认：
1. 代码修改已完成（所有编辑已保存）
2. 代码能通过基本的语法检查（无 SyntaxError）
3. 知道本次修改涉及的文件列表（通过 `git diff --name-only` 获取）

## 执行步骤

### Step 0: 收集变更范围

读取 `.catpaw/skills/post-code-guardian/SKILL.md`（本文件）获取编排规则。

执行以下命令获取本次变更的文件列表：

```bash
# 获取工作区所有变更（包括已暂存和未暂存的）
git diff --name-only HEAD
```

将变更文件分为三类：
- **源码文件**：`.py` / `.ts` / `.tsx` / `.js` / `.jsx`
- **文档文件**：`.md`
- **配置文件**：`.yaml` / `.yml` / `.toml` / `.json` / `.env`

如果**没有源码文件变更**，跳到 Step 4 输出"无源码变更，无需守护"。

### Step 1: 调度 code-review 技能

**技能路径**: `.catpaw/skills/code-review/SKILL.md`

**执行方式**: 读取该 SKILL.md 文件，按照其中定义的"执行步骤"对本次变更的源码文件逐一执行。

**检查维度**（按 code-review SKILL.md 定义）：
1. 收集上下文：阅读待检测文件及其依赖模块
2. 静态分析：逐函数检查逻辑正确性、边界条件、异常处理
3. 安全扫描：检查输入验证、认证授权、密钥管理、注入风险
4. 性能审查：识别阻塞操作、冗余计算、资源泄漏
5. 风格检查：命名规范、注释完整性、代码复杂度
6. 输出报告：按严重程度分级（Critical / Warning / Info）

**处理规则**:
- 如果发现 **Critical** 级别问题（安全漏洞、数据丢失风险等）→ **立即修复**，修复后重新检查
- 如果发现 **Warning** 级别问题（潜在 bug、性能隐患等）→ 修复，修复后继续
- 如果仅发现 **Info** 级别问题（命名风格等）→ 记录但不必修复，继续下一步
- 修复代码后，需重新确认修复未引入新问题（最多迭代 3 轮）

**输出**: code-review 报告，包含问题清单和修复记录

### Step 2: 调度 code-doc 技能

**技能路径**: `.catpaw/skills/code-doc/SKILL.md`

**执行方式**: 读取该 SKILL.md 文件，按照其中定义的"执行步骤"对本次变更的源码文件补充文档。

**文档补充范围**:
1. 为新增函数/类生成 Google 风格 docstring（Args / Returns / Raises）
2. 为修改了签名的函数更新 docstring
3. 为复杂逻辑（非显而易见的算法、异步流程、设计模式）添加行内注释
4. 为新增模块添加模块级文档字符串

**文档规范**（按 code-doc SKILL.md 定义）:
- Python: Google 风格 docstring
- TypeScript: JSDoc 风格注释
- 注释原则: Why > What，简洁精准，同步更新

**处理规则**:
- 仅对本次**变更的文件**补充文档，不批量修改未变更文件
- 已有完整 docstring 的函数不重复生成
- 禁止为简单赋值语句添加冗余注释
- 补充文档后需确认代码功能未改变（仅添加注释，不修改逻辑）

**输出**: code-doc 报告，列出新补充的 docstring 和注释位置

### Step 3: 调度 doc-sync 技能

**技能路径**: `.catpaw/skills/doc-sync/SKILL.md`

**执行方式**:
1. 读取 `.catpaw/skills/doc-sync/SKILL.md` 获取技能规则
2. 读取 `.catpaw/skills/doc-sync/doc_mapping.yaml` 获取代码-文档映射表
3. 对本次变更的源码文件，查找关联的 `.md` 文档
4. 逐文档执行一致性检查
5. 如发现不一致，**直接修改文档**使其与代码一致

**也可直接运行脚本**:
```bash
python .catpaw/skills/doc-sync/check_doc_sync.py --all
```

**检查维度**（按 doc-sync SKILL.md 定义）:
- 文件清单：文档中提到的文件路径是否仍然存在
- 函数/类签名：文档中引用的函数名、参数是否与代码一致
- 路由表：API 文档中的路由清单是否与 `@router` 装饰器一致
- 配置项：文档中列出的配置参数是否与 config.py 一致
- 进度状态：PROGRESS.md 中的模块完成状态是否反映最新代码
- Skills 清单：architecture/README.md 中的 Skills 表格是否完整

**处理规则**:
- 发现 Warning/Critical → **直接修改文档**使其一致
- 更新文档顶部的"最后更新"日期为当天
- 对已删除的规划文档（如 v2.0_improve.md 等）不做同步（这些描述未来目标，已清理）

**输出**: doc-sync 报告，包含检查结果和已更新的文档清单

### Step 4: 汇总输出守护报告

将三个子技能的输出汇总为一份统一的守护报告：

```markdown
# 🔒 Post-Code Guardian 守护报告

## 守护概要
- **执行时间**: YYYY-MM-DD HH:MM:SS
- **变更文件数**: N 个源码 + M 个文档
- **守护状态**: ✅ 全部通过 / ⚠️ 有遗留问题

## Step 1: Code Review（代码审查）
- **检查文件**: file1.py, file2.py, ...
- **发现问题**: Critical X / Warning Y / Info Z
- **已修复问题**: X 个
- **遗留问题**: 无 / [问题描述]
- **状态**: ✅ 通过

## Step 2: Code Doc（文档补充）
- **补充 docstring**: N 个函数 / M 个类
- **补充行内注释**: K 处
- **状态**: ✅ 完成

## Step 3: Doc Sync（文档同步）
- **检查文档**: doc1.md, doc2.md, ...
- **一致文档**: J 个
- **已更新文档**: K 个
  - ✏️ docs/architecture/L5-middleware.md — 补充参数说明
  - ✏️ docs/PROGRESS.md — 新增模块登记
- **状态**: ✅ 完成

## 守护结论
代码质量闭环已完成，所有变更文件已通过审查、文档补充、文档同步三重保障。
可以进行 git commit。
```

## 子技能调度顺序的必要性

| 顺序 | 技能 | 为什么在这个位置 |
|------|------|------------------|
| 1st | code-review | 必须最先执行，因为修复 bug 可能改变代码结构，后续步骤需基于修复后的代码 |
| 2nd | code-doc | 在 code-review 修复后执行，确保注释针对的是最终版代码，避免注释与代码不一致 |
| 3rd | doc-sync | 最后执行，因为前两步可能修改了代码签名/新增了文件，doc-sync 需要检查最终状态 |

## 与其他技能的关系

```
                     ┌─────────────────┐
                     │  rapid-dev      │ ← 开发阶段：快速生成脚手架代码
                     └────────┬────────┘
                              │ 代码修改完成
                              ▼
                     ┌─────────────────┐
                     │ post-code-      │ ← 收尾阶段：质量闭环守护
    ┌────────────────│ guardian (本技能)│────────────────┐
    │                └────────┬────────┘                │
    │                         │                         │
    ▼                         ▼                         ▼
┌───────────┐          ┌───────────┐          ┌───────────┐
│code-review│          │ code-doc  │          │ doc-sync  │
│ 代码审查   │          │ 文档补充   │          │ 文档同步   │
└───────────┘          └───────────┘          └───────────┘
                                                      │
                                                      ▼
                                              ┌─────────────┐
                                              │change-impact│ ← 可选：变更影响评估
                                              │  -report    │
                                              └─────────────┘
```

- `rapid-dev`：开发阶段使用，与本技能互斥（一个在"写代码"阶段，一个在"写完代码"阶段）
- `change-impact-report`：可在 doc-sync 之后可选调用，用于评估变更影响范围
- `fronted-design`：前端设计阶段使用，不纳入本编排流程
- `beginner-code-comment`：面向小白的注释风格，与 `code-doc` 互斥，不纳入本编排流程

## 常见陷阱

- **跳过 code-review 直接写文档**：如果代码有 bug，文档会描述错误的行为，导致双重错误
- **doc-sync 在 code-doc 之前执行**：code-doc 可能修改了函数签名注释，doc-sync 需要检查最终状态
- **对未变更的文件执行全量检查**：应仅检查本次 `git diff` 涉及的文件，避免范围蔓延
- **修复 code-review 问题后不重新检查**：修复本身可能引入新问题，需迭代验证（最多 3 轮）
- **守护报告中遗漏遗留问题**：如果有无法自动修复的问题，必须在报告中明确标注，不能静默跳过
