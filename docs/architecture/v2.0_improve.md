## 1. Multi-Agent升级：Supervisor + 专家群
### 1.1 目标架构
将当前单链管道升级为「Supervisor调度 + 专家Agent并行」多智能体架构：
1. 用户输入 → Supervisor（路由/编排，动态分派）
2. Supervisor并行分发至5类专家Agent（车控/导航/生活推荐/车辆健康/闲聊）
3. 所有专家输出汇总至Responder（统一回复生成、流式输出）
4. Responder结果流入Reviewer（质量检查 + 记忆存储）

### 1.2 与现状的衔接
1. Supervisor取代 PlannerAgent，接管意图路由、规划职责，决策能力从“单选技能”升级为“多选专家Agent，可并行分派”；
2. 专家Agent取代 ExecutorAgent + SkillOrchestrator，移除if-else分发逻辑，每个专家封装一组相关能力；
3. Responder/Reviewer 保留，统一汇总专家群输出；
4. 彻底删除 orchestrator 的if-else中间层，消除v1.0遗留分支债务。

### 1.3 LangGraph 落地实现
采用LangGraph官方multi-agent pattern Supervisor模式，核心代码逻辑：
1. 新建文件 `agent/supervisor_graph.py`，导入LangGraph核心组件（END、StateGraph等）；
2. 定义全局工作流、全局状态 SupervisorState；
3. 注册节点：Supervisor路由节点、5类专家节点、Responder、Reviewer；
4. 流转规则：专家完成后回流Supervisor支持多轮分派；Reviewer执行完毕走向END；
5. 配套能力：
    - 并行分派：多专家asyncio.gather同时执行，汇总输出；
    - 回淌分派：专家完成回流Supervisor，二次拆分任务；
    - checkpoint持久化：SqliteSaver（开发+生产通用），thread_id=session_id，状态落盘 SQLite 文件，重启不丢失。

### 1.4 专家Agent设计（共5个独立专家）
所有专家独立类，实现`async def run(state) -> ExpertResult`，内部调用SkillRegistry，支持并行执行：
| 专家名称 | 封装技能 | 输出内容 |
| ---- | ---- | ---- |
| 车控专家 VehicleExpert | climate/window/seat/media/status | 车控执行结果 |
| 导航专家 NavExpert | navigation + 路线规划 + 兴趣点搜索 | 导航指令、路线信息 |
| 生活推荐专家 LifestyleExpert | order_food + 本地生活推荐(新增) + web_search | 推荐结果 |
| 车辆健康专家 HealthExpert | 车辆诊断 + 保养提醒 + 故障码翻译(新增) | 诊断报告 |
| 闲聊专家 ChatExpert | 纯LLM闲聊、知识库问答 | LLM回复 |

## 2. 技能扩展：4个新方向
原有9个技能，扩展至18个，新增4大类技能组，配套改造技能注册机制。
### 2.1 车主习惯画像（Habit Profile）
基于现有`get_user_profile()`扩展，学习车主习惯实现主动推荐，Neo4j新增HABIT关系存储习惯三元组。
新增技能：
1. habit_record：用户表述偏好，提取习惯写入Neo4j HABIT关系；
2. habit_recommend：早晨上车/导航启动，查询图谱习惯主动推荐；
3. habit_adjust：读取画像批量下发车控指令。
主动推荐触发：Reviewer节点新增ProactiveHook，检测到早晨对话/导航启动时异步推送推荐，不阻塞主响应。

### 2.2 日程/提醒类（Schedule & Reminder）
支持定时提醒、查询、取消，存储复用Redis，采用Sorted Set按时间排序；后台Celery定时扫描，到期WebSocket推送消息。
新增技能：
1. set_reminder：解析时间+内容，持久化存储提醒；
2. query_reminder：查询用户全部待办提醒；
3. cancel_reminder：删除指定提醒。

### 2.3 车辆健康/诊断（Vehicle Health）
车辆故障解读、故障码翻译、保养周期计算，数据存入Cherry知识库，文档型RAG检索。
新增技能：
1. diagnose_vehicle：车辆异常问题解读，调取车辆状态；
2. decode_dtc：故障码释义，查询故障知识库；
3. maintenance_advice：根据里程/时间生成保养建议。

### 2.4 本地生活/推荐（Local Life）
多轮上下文推荐，复用图谱POI检索能力，新增多轮槽位管理。
新增技能：
1. recommend_poi：周边餐饮/景点检索+距离排序；
2. multi_turn_refine：多轮填充推荐槽位，保留上下文；
3. preference_filter：基于用户偏好筛选候选推荐结果。

### 2.5 技能注册机制改造
现状：技能硬编码在SkillRegistry初始化函数；
改造：装饰器自动注册，新增技能仅添加装饰器，无需修改registry代码，实现插件化：
1. 定义全局注册表`_REGISTRY`与装饰器`register_skill`；
2. 所有技能类添加`@register_skill`装饰器自动入库；
3. SkillRegistry初始化自动遍历注册表完成注册；
4. 配合专家分组配置，实现技能归属专家的灵活配置。

## 3. RAG升级：GraphRAG深化 + Cherry知识库
双知识库分层设计，统一Retriever路由分发，两类知识库各司其职。
### 3.1 双知识库定位区分
| 知识库 | 用途 | 数据类型 | 检索方式 |
| ---- | ---- | ---- | ---- |
| GraphRAG（现有深化） | 用户记忆、习惯画像、关系网络 | 结构化三元组 | 向量+图谱遍历+RRF融合 |
| Cherry知识库（新增） | 车手册、故障码、FAQ、保养规范 | 长文档 | 文档分块+向量+全文检索 |

### 3.2 GraphRAG 现有能力深化
基于原有RRF融合逻辑，新增4项能力：
1. 混合检索：向量语义、BM25全文、图谱关系三路检索，RRF融合（原仅两路）；
2. 查询改写：Supervisor路由新增QueryRewriter，LLM补全上下文生成完整查询；
3. Rerank：本地部署 BAAI/bge-reranker-v2-m3 模型（免费，约560MB），Top20结果重排至Top5，提升检索精度；
4. 引用溯源：每条检索结果携带source/score，回复中标注数据来源。

### 3.3 Cherry知识库接入（全新模块）
选型：基于现有Milvus新建文档集合，不引入新向量库组件；
完整链路：文档分块 → Embedding向量化 → Milvus文档向量集合存储 → KBRetriever检索；
新增统一路由层 `rag/unified_retriever.py`：
根据query_type区分分发至GraphRAG（记忆/习惯）、Cherry知识库（手册/故障）、混合检索，Supervisor标记query_type实现统一入口。

### 3.4 知识库管理API
复用现有admin路由，新增知识库管理接口，前端settings页面增加知识库管理模块：
1. POST /admin/kb/upload：上传文档（复用OSS存储）；
2. POST /admin/kb/reindex：重建向量索引；
3. GET /admin/kb/stats：知识库容量/文档统计。

## 4. 上下文记忆 + Prompt工程改进
### 4.1 记忆分层增强
修复现有summary跨请求丢失问题，分层管理用户记忆：
1. running_summary持久化：绑定checkpoint，随会话state永久存储，实现滚动摘要；
2. 记忆类型分层：Neo4j区分三类关系：PREFERENCE（长期偏好）、HABIT（习惯带时间）、CONTEXT（短期对话），召回按时间加权；
3. 记忆衰减：长期记忆降低权重，Neo4j增加last_accessed属性，召回按「新鲜度×频次」排序。

### 4.2 Prompt工程改进
修复硬编码提示词、字数限制过短问题：
1. 模板外置：新建`nexus/prompts/`目录，按场景拆分md模板文件（规划、闲聊、搜索、车控、记忆提取）；
2. 动态变量注入：模板支持`{user_profile}` `{memory}` `{slots}`运行时填充；
3. Few-shot示例：每个技能内置example样本，路由prompt自动注入；
4. Prompt版本管理：模板带版本号，支持A/B测试评估效果。

### 4.3 上下文窗口管理
1. Token估算：从简单长度换算升级为tiktoken精准计数；
2. 动态上下文上限：max_context_tokens不再硬编码1600，按模型窗口动态计算，预留30%回复余量。

## 5. LangGraph 使用改进 + Redis缓存重设计
### 5.1 LangGraph 现有问题与改造
| v1.0问题 | v2.0改进方案 |
| ---- | ---- |
| stream() 手写循环，不标准 | astream_events(v2)统一驱动，图结构为唯一事实来源 |
| AgentState无reducer，全量返回state | history用add_messages，metadata字典合并reducer，仅增量更新 |
| 无持久化checkpoint | SqliteSaver（SQLite 文件落盘，无需额外容器，适合个人开发者），thread_id绑定session |
| 无前端进度事件 | astream_events透出规划/执行/生成阶段，前端展示进度 |
| 无循环保护 | Supervisor回流预设recursion_limit=10，防止死循环 |

### 5.2 Redis缓存重设计（核心性能优化）
现状：全量遍历所有缓存条目计算相似度O(n)，高并发性能崩溃；
改造：基于Redis Stack Redisearch + VECTOR向量索引实现KNN向量检索O(log n)；
缓存增强策略：
1. 按user_id分片索引，跨用户缓存隔离；
2. 车控指令不缓存：执行时判断action类型，车控类跳过缓存，规避安全风险；
3. TTL分级：闲聊缓存1h、知识库问答24h、车控指令永不缓存；
4. 缓存预热：高频查询启动预加载；
5. 分布式锁：防止同一查询并发击穿缓存。

### 5.3 缓存与Agent解耦
缓存检查从chat.py路由层上移至Supervisor前置CacheGuard节点：缓存命中直接短路返回Responder，未命中才进入Supervisor调度，分离缓存与业务Agent逻辑。

## 8. 优先级执行顺序（资源有限时按产出比排序）
1. Supervisor+专家群：整体架构骨架，所有扩展依赖此底座，优先开发；
2. 技能装饰器注册 + 四大类新技能：用户感知最强，直接丰富产品能力；
3. Redis VECTOR向量缓存：解决核心性能瓶颈，收益立竿见影；
4. 前端交互补完 + 闲置依赖接入：修复静态页面问题，产品完整可用；
5. GraphRAG深化 + Cherry知识库：提升问答、检索智能底层能力；
6. 记忆分层 + Prompt工程：上层体验优化，锦上添花。

## 9. 风险与注意事项
1. Supervisor回流死循环：必须配置recursion_limit，专家互相分派需要明确终止条件；
2. 多专家并行车控冲突：导航/车控可并行，多个车控指令需要Supervisor层冲突仲裁；
3. 知识库性能：Cherry文档库文档多时检索变慢，需分块策略、索引优化；
4. 缓存与车控安全：严格执行车控指令永不缓存，避免缓存旧指令导致车辆误操作；
5. Prompt注入安全：用户输入LLM路由隔离清洗，习惯画像注入prompt同样做过滤防注入；
6. 落地规范：每阶段完成后做联调验证，避免架构漂移，所有改动均提供代码文件行号落地路径。

## 10. 技术选型决策与外部资源清单
### 10.1 Rerank 模型（免费方案）
- 模型：BAAI/bge-reranker-v2-m3
- 下载地址：https://huggingface.co/BAAI/bge-reranker-v2-m3
- 存放路径：`./models/reranker/bge-reranker-v2-m3/`
- 推理框架：sentence-transformers（需新增依赖 `sentence-transformers>=2.7.0`）
- 显存/内存：约560MB，CPU 可推理（延迟约200ms/20条），GPU 更快
- 备选（纯Python无模型）：rank-bm25 仅做 BM25 召回路，不涉及 rerank

### 10.2 Checkpoint 持久化
- 选型：SqliteSaver（langgraph-checkpoint-sqlite）
- 理由：个人开发者展示技术，SqliteSaver 无需额外容器（PostgreSQL），单个 .db 文件即可持久化，重启不丢失，部署成本为零
- 依赖：`langgraph-checkpoint-sqlite>=2.0.0`（需新增）
- 存储路径：`./data/checkpoints/nexus_checkpoints.db`
- 开发环境可切换 MemorySaver 做快速调试（配置项控制）

### 10.3 前端 3D 车辆模型
- 选型：@react-three/fiber + @react-three/drei 程序化建模
- 理由：无需下载外部 GLB/GLTF 模型文件，代码生成低多边形车体，体积小、可定制、科技感强
- 新增依赖：three、@react-three/fiber、@react-three/drei
- 交互：点击车身部位（车窗/座椅/空调）高亮并发送对应车控指令

| 项目 | 改动内容 | 说明 |

| Redis 镜像 | docker-compose.yml `redis:7.2-alpine` → `redis/redis-stack-server:7.2.0-v4` | 支持 RediSearch + VECTOR 向量索引 |
| Celery Worker | docker-compose.yml 新增 `celery-worker` 服务 | 日程提醒定时扫描，RabbitMQ 已就绪 |
