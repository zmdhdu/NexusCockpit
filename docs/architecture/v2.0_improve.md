# NexusCockpit v2 扩展能力方案
## 一、方案基础信息
### 定位
仅聚焦扩展能力，不涉及v1.0已知bug修复（安全分级、并发竞态、双写一致性、文档乱码等另行处理）。
### 目标
把v1.0线性四阶段Agent+9个技能，升级为**基础GraphRAG+遍历式缓存+静态前端** → **Supervisor专家群+多技能生态+双知识库+向量缓存+科技感UI**。

## 三、核心六大改造模块
### 1. Multi-Agent升级：Supervisor + 专家群
#### 1.1 目标架构
废弃单链管道，改为**Supervisor调度 + 多专家Agent并行**架构：
1. 用户输入 → Supervisor（路由/编排）
2. Supervisor动态分发任务至5类独立专家Agent，可并行执行
3. 各专家输出统一汇总至Responder（流式回复生成）
4. Reviewer统一质量校验、持久化记忆

#### 1.2 和现有代码衔接改造点
1. Supervisor替代原plannerAgent，意图路由升级为专家分发，支持多专家并行
2. 专家Agent取代ExecutorAgent，废弃skillOrchestrator的if-else分支，每个专家封装一组技能，内部可并行调用多个技能
3. 保留Responder/Reviewer，统一聚合专家输出，不再单技能结果返回
4. 删除旧orchestrator中间if-else分发逻辑，修复v1.0 Intent全包问题

#### 1.3 LangGraph落地实现
采用官方Supervisor多智能体模式，新增`agent/supervisor_graph.py`：
1. 节点：supervisor路由节点 + 5类专家节点 + responder回复 + reviewer校验
2. 流转逻辑：
   - Supervisor可分发至任意专家；专家完成回流Supervisor，支持多轮递归
   - 条件分支：无更多任务则走Responder，最终Reviewer后结束流程
3. 关键能力：
   - 并行分发：asyncio.gather同时调用多个专家，解决v1.0串行执行限制
   - 回流再分派：专家完成后可二次分发，支持多轮复杂任务
   - checkpoint持久化：thread_id+session_id会话隔离，修复v1.0进程内存dict丢失问题

#### 1.4 5类专家Agent设计（独立可并行）
| 专家 | 封装技能 | 输出 |
| ---- | ---- | ---- |
| 车控专家 VehicleExpert | climate/window/seat/media/status | 车控执行结果 |
| 导航专家 NavExpert | navigation、路线规划、兴趣点搜索 | 导航指令+路线信息 |
| 生活推荐专家 LifestyleExpert | order_food、本地生活推荐、web_search | 推荐结果 |
| 车辆健康专家 HealthExpert | 车辆诊断、保养提醒、故障码翻译 | 诊断报告 |
| 闲聊专家 ChatExpert | 纯LLM闲聊、知识库问答 | LLM回复 |

每个专家实现`async def run(state) -> ExpertResult`，内部通过SkillRegistry调度技能。

### 2. 技能扩展：4大新增方向（9→18个技能）
#### 2.1 车主习惯画像（Habit Profile）
- 目标：学习车主习惯，主动推荐；复用现有`get_user_profile()`
- 新增技能：`habit_record`
  - 触发：用户表达偏好（如“我喜欢24度，常去公司”）
  - 实现：提取习惯三元组写入Neo4j HABIT关系
- 联动：Reviewer新增ProactiveHook，检测到高频场景自动推送习惯推荐

#### 2.2 日程/提醒类（Schedule & Reminder）
- 新增3个技能：
  1. `set_reminder`：解析时间+内容，持久化存储提醒
  2. `query_reminder`：查询用户待办提醒
  3. `cancel_reminder`：删除指定提醒
- 存储：Redis SortedSet按时间排序，`nexus:reminders:{user_id}`
- 推送机制：后台Celery定时任务/asyncio定时扫描，到点WebSocket推送

#### 2.3 车辆健康/诊断（Vehicle Health）
- 新增3个技能：
  1. `diagnose_vehicle`：解析车辆故障灯，读取车辆状态解读
  2. `decode_dtc`：查询故障码知识库解释故障含义
  3. `maintenance_advice`：根据里程/时间生成保养建议
- 知识来源：OBD故障码、保养周期存入Cherry文档知识库，RAG检索

#### 2.4 本地生活/推荐（Local Life）
- 新增3个技能，复用图谱POI检索`graph_store.search_food()`
  1. `recommend_poi`：附近餐饮/景点，图谱检索+距离排序
  2. `multi_turn_refine`：多轮上下文保留，支持换一批、筛选
  3. `preference_filter`：基于用户习惯筛选候选推荐结果
- 上下文改造：AgentState新增`recommendation_slots`字段，存储多轮推荐上下文

#### 2.5 技能注册机制重构
废弃硬编码`SkillRegistry.__init__`遍历注册，改为**装饰器自动发现**：
1. 提供`@register_skill`装饰器，技能类直接标记即可自动注册
2. 专家Agent仅需配置技能分组，实现插件化扩展，无需修改注册表代码

### 3. RAG升级：GraphRAG深化 + Cherry双知识库
#### 3.1 双知识库分层定位
| 知识库 | 用途 | 数据类型 | 检索方式 |
| ---- | ---- | ---- | ---- |
| GraphRAG（现有深化） | 用户记忆、习惯画像、人物关系 | 结构化三元组 | 向量+图谱遍历+RRF融合 |
| Cherry知识库（新增） | 车手册、故障码、FAQ、保养规范 | 长文文档 | 文档分块+向量+全文检索 |

#### 3.2 GraphRAG 现有能力深化
1. 混合检索：向量语义+BM25全文+Neo4j图谱三路RRF融合
2. 查询改写：Supervisor节点增加QueryRewriter，LLM优化用户原始查询
3. Rerank精排：召回Top20后使用rerank模型筛选Top5，提升精准度
4. 引用溯源：每条检索结果携带source/score，回复标注信息来源增强可信度

#### 3.3 Cherry知识库接入（新增）
1. 选型：Milvus新建`kb_docs`向量集合，复用现有Embedding服务，不新增组件
2. 文档处理链路：文档分块（512块长、64重叠）→ Embedding入库Milvus
3. 统一Retriever路由层`unified_retriever.py`：
   - 根据`query_type`分发：memory（GraphRAG）/knowledge（Cherry）/混合检索
   - Supervisor路由携带query_type，一套入口统一管理两类知识库

#### 3.4 知识库管理API
新增`/admin/kb`管理接口，复用OSS存储：
- POST `/admin/kb/upload`：上传文档
- POST `/admin/kb/reindex`：重建向量索引
- GET `/admin/kb/stats`：知识库容量统计
前端settings页面新增知识库管理可视化区块

### 4. 上下文记忆 + Prompt工程改进
#### 4.1 记忆分层增强
1. running_summary持久化：接入checkpoint，摘要随会话状态落盘，解决跨请求丢失
2. 记忆类型分层：Neo4j区分PREFERENCE偏好、HABIT习惯、CONTEXT短期上下文
3. 记忆衰减：长期低频记忆降低权重，检索按「新鲜度+访问频次」排序

#### 4.2 Prompt工程标准化改造
1. 模板外置：新建`nexus/prompts/`目录，按场景拆分md模板文件（车控、闲聊、检索、提取等）
2. 动态变量注入：模板支持`user_profile`/`memory`/`slots`运行时填充
3. Few-shot内置：每个技能绑定示例，自动注入路由，提升意图识别准确率
4. 版本管理：模板版本号，A/B测试对比效果，配套评估指标

#### 4.3 上下文窗口精准管控
1. token估算：替换粗糙len/1.8估算，使用tiktoken精确计数
2. 动态上下文上限：max_context_tokens按模型窗口动态分配，预留30%回复余量

### 5. LangGraph优化 + Redis缓存重设计
#### 5.1 LangGraph v1.0问题修复
| v1.0现存问题 | v2.0优化方案 |
| ---- | ---- |
| stream() 手写循环 | astream_events(v2)统一驱动，唯一事件来源 |
| AgentState无reducer，节点覆盖全量state | history节点用dict合并更新，增量更新状态 |
| 无checkpoint持久化 | MemorySaver会话持久，thread_id隔离 |
| 无阶段事件输出 | astream_events输出规划/执行/生成状态，前端展示 |
| 无递归上限，死循环风险 | Supervisor预设recursion_limit=10拦截循环 |

#### 5.2 Redis缓存架构重构（核心性能优化）
原逻辑缺陷：全量Hash遍历计算相似度O(n)，并发性能崩塌
升级为Redis Stack VECTOR向量索引：
1. 索引结构：`nexus:cache`向量索引，COSINE距离相似度检索
2. 查询优化：KNN向量检索，复杂度O(log n)
3. 缓存策略增强：
   - 用户分片：索引携带user_id过滤，避免全表遍历
   - 车控指令隔离：新增`has_side_effect`标记，车控操作禁止缓存，规避安全风险
   - TTL分级：闲聊缓存1h，知识库缓存24h，车控永不缓存
   - 预热缓存：高频场景（天气、车辆状态）启动预加载
   - 击穿防护：Redis分布式锁防并发穿透

#### 5.3 缓存与Agent流程解耦
新增CacheGuard前置节点（Supervisor之前）：
1. 命中缓存：直接短路返回Responder，不走完整Agent流程
2. 未命中：正常进入Supervisor调度
优势：缓存逻辑统一管控，可观测性更强

## 六、风险与注意事项
1. Supervisor死循环：必须配置`recursion_limit`，专家互相分发需设计终止条件
2. 多专家车控冲突：并行分发同时触发车控指令时，Supervisor增加冲突仲裁（导航+调温可并行，双车控指令互斥排队）
3. 缓存车控安全风险：必须给车控操作添加`has_side_effect`标记，禁止缓存，避免重复执行硬件指令
4. Prompt注入防护：用户输入LLM路由层做清洗拦截，习惯画像注入Prompt同样做过滤
5. 交付规范：每个阶段完成后单独验证调试验证，防止架构漂移
