## 一、背景与动机

### 1.1 v2.0 现状

v2.0 是一个**单座舱**车载语音 Agent 系统：

- Supervisor + 5 Expert Agents 并行调度
- GraphRAG 三路融合检索
- Redis 语义缓存 + 限流
- 前端 4 页面（仪表盘/聊天/车控/设置）

### 1.2 v2.1 要解决的问题

| 问题 | v2.0 现状 | v2.1 目标 |
|------|----------|----------|
| 单座舱局限 | 只有一个用户、一套车控 | 多座舱并行，每个座舱独立用户/车控 |
| 缺乏集中观测 | 只有单座舱仪表盘 | 数据中台看板，观测所有座舱使用情况与并发能力 |
| 中间件不可见 | 中间件状态分散在健康检查 | 专门的中间件看板，展示各座舱缓存/队列隔离状态 |
| Agent 监控缺失 | 无主动状态检测 | SubAgent 不定时监控，主 Agent 二次确认 |
| 扩展性不足 | 座舱硬编码 | 设置界面注册座舱用户，动态扩展 |

### 1.3 设计理念

参考 **Claude Agent 框架**的思想：

```
SubAgent（子代理）不定时监控检测各座舱状态
    ↓ 调用 LLM 判断是否出现异常
    ↓ 将判断结果传给主 Agent
主 Agent（主代理）二次检测确认
    ↓ 确认无问题后
    ↓ 将该座舱的执行结果返回
```

这种 **SubAgent → 主 Agent 双层确认** 机制确保：
1. **早期发现**：SubAgent 主动巡检，不依赖用户触发
2. **降低误报**：主 Agent 二次确认避免 SubAgent 误判
3. **安全回传**：只有主 Agent 确认 OK 的结果才会返回给用户

## 二、v2.0 → v2.1 核心变化

```
v2.0:  单座舱 / Supervisor → Experts → Responder → Reviewer
                    ↓
v2.1:  多座舱 CS 架构
       ├── 座舱1 (cockpit-01) ──┐
       ├── 座舱2 (cockpit-02) ──┤── 数据中台（观测/统计/并发监控）
       ├── 座舱3 (cockpit-03) ──┘
       │
       ├── SubAgent Monitor Layer（不定时巡检 + LLM 判断）
       │      ↓
       ├── MainAgent Confirm Layer（二次确认 + 安全回传）
       │
       ├── 中间件隔离层（Redis DB 分区 / RabbitMQ vhost / Milvus collection）
       │
       └── 设置中心（座舱注册 / 中间件配置 / 用户管理）
```

### 变化对照表

| 维度 | v2.0 | v2.1 |
|------|------|------|
| 座舱数量 | 1 | 3（可扩展到 N） |
| 架构模式 | 单体 | CS 多客户端 + 集中服务端 |
| Agent 层 | Supervisor + 5 Experts | SubAgent 监控层 + MainAgent 确认层 + 原有 Experts |
| 数据隔离 | 全局共享 | 按座舱 ID 隔离（Redis DB / Milvus collection / MySQL schema） |
| 前端页面 | 4 页面 | 侧栏分组：座舱1/2/3 + 数据中台 + 中间件 + 设置 |
| 可观测性 | Prometheus + Grafana | 新增数据中台业务看板 + 中间件状态看板 |
| 用户管理 | JWT 单用户 | 座舱用户注册 + JWT 多用户 + RBAC |

### 4.2 各中间件隔离方案

#### Redis（语义缓存 + 限流 + 会话）

```python
# === 正式方案：座舱 ID → Redis DB 编号映射 ===
COCKPIT_REDIS_DB_MAP = {
    "cockpit-01": 1,
    "cockpit-02": 2,
    "cockpit-03": 3,
    # 公共配置/全局统计使用 DB 0
}

# 每个座舱拥有独立的 Redis DB，缓存/限流/会话互不干扰
# key 前缀也加上 cockpit_id 做双重隔离:
#   cache:cockpit-01:{hash}
#   ratelimit:cockpit-01:{user_id}
#   session:cockpit-01:{session_id}
```

> **Demo 简化方案**（默认启用，`COCKPIT_ISOLATION_MODE=shared`）：
> 单 Redis DB（DB 0）+ key 前缀隔离（`cockpit-{id}:`），3 个座舱数据量小，不会冲突。
> 正式环境切换为 `COCKPIT_ISOLATION_MODE=strict` 即启用 DB 分区隔离。
> 两种模式通过 `isolation_mode` 配置项切换，代码自动适配。

#### Milvus（向量检索）

```python
# 每个座舱独立的 collection
# 命名规则: cockpit_{id}_{purpose}
collections = {
    "cockpit-01": {
        "memory": "cockpit_01_memory",   # 用户记忆
        "food": "cockpit_01_food",       # 食物知识库
    },
    "cockpit-02": {
        "memory": "cockpit_02_memory",
        "food": "cockpit_02_food",
    },
    # ...
}
```

**Demo 简化方案**：共享 collection + `cockpit_id` 字段过滤，减少 collection 数量。

#### Neo4j（知识图谱）

```cypher

CREATE (u:User {cockpit_id: "cockpit-01", name: "张三"})

// 查询时必须带 cockpit_id 过滤

```

#### MySQL（用户数据 + 审计日志）

```sql
-- 所有业务表添加 cockpit_id 列 + 索引

ALTER TABLE users ADD INDEX idx_cockpit (cockpit_id);

ALTER TABLE chat_history ADD COLUMN cockpit_id VARCHAR(32) NOT NULL DEFAULT 'cockpit-01';
ALTER TABLE chat_history ADD INDEX idx_cockpit (cockpit_id);
```

```python
1. 每个 SubAgent 负责一个座舱
2. 巡检频率可配置（默认 30-60s，加随机抖动避免同步）
3. 巡检结果通过 Redis Pub/Sub 上报给 MainAgent
4. 不阻塞用户主请求路径（异步后台运行）

        llm_client: AsyncOpenAI,

        self.llm_client = llm_client
        self.check_interval = check_interval
        self._running = False
```

### 6.2 座舱管理器 (CockpitManager)

```python
# backend_design/nexus/core/cockpit_manager.py

"""
座舱管理器 — 管理所有座舱的注册、查询、状态

核心职责:
1. 座舱注册（设置界面创建新座舱）
2. 座舱查询（路由层根据 cockpit_id 查询座舱配置）
3. 座舱状态（SubAgent 监控层查询座舱健康状态）
4. 座舱隔离（为每个座舱分配 Redis DB / Milvus collection 等）
"""

from dataclasses import dataclass, field
from typing import Optional
from datetime import datetime
import uuid

@dataclass
class CockpitConfig:
    """单个座舱的配置。"""
    cockpit_id: str                    # 唯一标识: cockpit-01
    name: str                          # 显示名称: "座舱1"
    user_id: str                       # 绑定用户 ID
    vehicle_adapter: str = "mock"      # 车控适配器: mock/http/mcp
    redis_db: int = 0                  # Redis DB 编号（隔离）
    milvus_collection_prefix: str = "" # Milvus collection 前缀
    created_at: datetime = field(default_factory=datetime.now)
    is_active: bool = True             # 是否启用

class CockpitManager:
    """座舱管理器单例。"""

    def __init__(self):
        self._cockpits: dict[str, CockpitConfig] = {}
        self._next_seq: int = 0        # 自增序号（不随删除递减）
        self._init_default_cockpits()

    def _init_default_cockpits(self):
        """初始化 3 个默认座舱。"""
        for i in range(1, 4):
            cid = f"cockpit-0{i}"
            self._cockpits[cid] = CockpitConfig(
                cockpit_id=cid,
                name=f"座舱{i}",
                user_id=f"user_0{i}",
                redis_db=i,
                milvus_collection_prefix=f"cockpit_0{i}",
            )
        self._next_seq = 4  # 下一个注册的序号从 4 开始

    def get_cockpit(self, cockpit_id: str) -> Optional[CockpitConfig]:
        return self._cockpits.get(cockpit_id)

    def list_cockpits(self) -> list[CockpitConfig]:
        return list(self._cockpits.values())

    def register_cockpit(
        self,
        name: str,
        user_id: str,
        vehicle_adapter: str = "mock",
        **kwargs,
    ) -> CockpitConfig:
        """注册新座舱（设置界面调用）。

        使用 _next_seq 自增序号生成 ID，避免删除后 len() 导致 ID 冲突。
        可选参数仅接受 CockpitConfig 的合法字段。
        """
        idx = self._next_seq
        cid = f"cockpit-{idx:02d}"

        # 确保不与已存在的 cockpit_id 冲突（极端情况）
        while cid in self._cockpits:
            self._next_seq += 1
            idx = self._next_seq
            cid = f"cockpit-{idx:02d}"

        config = CockpitConfig(
            cockpit_id=cid,
            name=name,
            user_id=user_id,
            vehicle_adapter=vehicle_adapter,
            redis_db=idx,
            milvus_collection_prefix=f"cockpit_{idx:02d}",
            **kwargs,
        )
        self._cockpits[cid] = config
        self._next_seq += 1  # 序号递增，不随删除回退
        return config

    def unregister_cockpit(self, cockpit_id: str) -> bool:
        """注销座舱（软删除，标记 is_active=False）。序号不回退。"""
        config = self._cockpits.get(cockpit_id)
        if config:
            config.is_active = False
            return True
        return False
```

### 6.3.1 `cockpit_id` 与现有 `user_id` 的关系及迁移方案

> **核心问题**：v2.0 使用 `user_id` 做 Redis 缓存隔离（RediSearch TAG 字段）和 Session 隔离。v2.1 引入 `cockpit_id` 后，两者如何共存？

#### 关系定义

```
cockpit_id (座舱 ID)
  ├── 1 个座舱可绑定 1 个或多个 user_id（RBAC: cockpit_admin 可管理多个座舱）
  ├── 隔离粒度: 中间件层（Redis DB / Milvus collection / MySQL 行级）
  └── 来源: JWT Token + URL Path /cockpit/{cockpit_id}/

user_id (用户 ID)
  ├── 隔离粒度: Redis 缓存分片（RediSearch TAG）+ Session 历史
  └── 来源: JWT Token
```

| 隔离维度 | v2.0 隔离键 | v2.1 隔离键 | 说明 |
|---------|-----------|-----------|------|
| Redis 语义缓存 | `user_id` TAG | `cockpit_id` TAG（主）+ `user_id` TAG（辅） | 同一座舱不同用户的缓存共享，跨座舱隔离 |
| Redis 会话 | `session:{user_id}` | `session:{cockpit_id}:{user_id}` | 前缀加 cockpit_id |
| Redis 限流 | `ratelimit:{user_id}` | `ratelimit:{cockpit_id}:{user_id}` | 座舱级令牌桶 |
| Milvus 向量 | `user_id` 字段过滤 | `cockpit_id` 字段过滤 | 按座舱隔离记忆库 |
| MySQL | 无 | `cockpit_id` 列 | 行级隔离 |

#### 迁移方案

### 6.4 API 路由设计

> **路由归属说明**：v2.1 引入 Go 网关后，API 分为两类：
> - **Go 原生处理**：无需 AI 计算的 CRUD/查询/状态接口，Go 直接查中间件返回（性能极高）
> - **Go 转发 Python**：需要 Agent/RAG/ASR/TTS 的接口，Go 完成鉴权+限流后转发给 Python AI 服务

#### 座舱路由（混合：Go 原生 + Go 转发 Python）

```python
# --- Go 原生处理（无需 AI）---

# GET  /cockpit/{cockpit_id}/status
#      → Go 查 Redis 缓存 + MySQL → 返回座舱状态 + 车辆状态
#      → 实现: nexus_gate/internal/handler/cockpit.go

# GET  /cockpit/{cockpit_id}/vehicle/status
#      → Go 查 Redis 缓存 → 返回车辆状态
#      → 实现: nexus_gate/internal/handler/vehicle.go

# --- Go 转发 Python（需要 AI）---

# POST /cockpit/{cockpit_id}/chat
#      → Go 鉴权+限流 → 转发 Python: Agent 工作流
#      → Python 实现: nexus/api/routes/cockpit.py

# POST /cockpit/{cockpit_id}/chat/stream
#      → Go 鉴权 → 转发 Python: Agent 流式（gRPC stream / HTTP SSE）
#      → Python 实现: nexus/api/routes/cockpit.py

# POST /cockpit/{cockpit_id}/vehicle/cmd
#      → Go 鉴权+限流 → 转发 Python: 车控技能执行
#      → Python 实现: nexus/api/routes/cockpit.py

# POST /cockpit/{cockpit_id}/asr
#      → Go 转发 Python: FunASR 语音识别
#      → Python 实现: nexus/api/routes/cockpit.py

# POST /cockpit/{cockpit_id}/tts
#      → Go 转发 Python: CosyVoice 语音合成
#      → Python 实现: nexus/api/routes/cockpit.py

# WS   /cockpit/{cockpit_id}/ws/chat
#      → Go 管理 WebSocket 连接 → gRPC 调 Python: Agent 流式
#      → Go 实现: nexus_gate/internal/ws/hub.go
#      → Python 实现: nexus/api/websocket.py
```

#### 数据中台路由（Go 原生处理）

```go
// nexus_gate/internal/handler/dataplatform.go
// 所有数据中台 API 由 Go 直接查 MySQL/Prometheus 返回，无需 Python

// GET  /dataplatform/overview              → Go 查 MySQL cockpit_stats → 全局概览
// GET  /dataplatform/cockpit/{cockpit_id}  → Go 查 MySQL → 单座舱详情

// GET  /dataplatform/agent/activity        → Go 查 MySQL subagent_logs → Agent 活动
// WS   /dataplatform/ws/realtime           → Go 实时推送指标变化
```

#### 中间件状态路由（Go 原生处理）

```go
// nexus_gate/internal/handler/middleware.go
// 所有中间件状态 API 由 Go 直接查询各中间件返回

// GET  /middleware/redis/status            → Go 调 Redis INFO → 各座舱 DB 状态
// GET  /middleware/milvus/status           → Go 调 Milvus Management API → collection 状态
```

#### 设置路由（Go 原生处理）

```go
// nexus_gate/internal/handler/settings.go
// 座舱/用户/配置 CRUD 由 Go 直接操作 MySQL

// GET  /settings/cockpits                  → Go 查 MySQL cockpits 表 → 座舱列表
```

### 6.6 数据库表设计

```sql
-- 座舱表
CREATE TABLE cockpits (
    cockpit_id   VARCHAR(32) PRIMARY KEY,
    name         VARCHAR(64) NOT NULL,
    user_id      VARCHAR(64) NOT NULL,
    vehicle_adapter VARCHAR(32) DEFAULT 'mock',
    redis_db     INT DEFAULT 0,
    is_active    BOOLEAN DEFAULT TRUE,
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 座舱使用统计表（数据中台用）
CREATE TABLE cockpit_stats (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    cockpit_id   VARCHAR(32) NOT NULL,
    stat_time    DATETIME NOT NULL,
    chat_count   INT DEFAULT 0,
    vehicle_cmd_count INT DEFAULT 0,
    cache_hits   INT DEFAULT 0,
    cache_misses INT DEFAULT 0,
    avg_latency_ms FLOAT DEFAULT 0,
    p95_latency_ms FLOAT DEFAULT 0,
    error_count  INT DEFAULT 0,
    INDEX idx_cockpit_time (cockpit_id, stat_time)
);

-- SubAgent 巡检日志
CREATE TABLE subagent_logs (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    cockpit_id   VARCHAR(32) NOT NULL,
    check_time   DATETIME NOT NULL,
    check_items  JSON,           -- 巡检项快照
    llm_judgment JSON,           -- LLM 判断结果
    is_anomaly   BOOLEAN DEFAULT FALSE,
    INDEX idx_cockpit_time (cockpit_id, check_time)
);

-- MainAgent 确认日志
CREATE TABLE mainagent_logs (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    cockpit_id   VARCHAR(32) NOT NULL,
    alert_time   DATETIME NOT NULL,
    alert_type   VARCHAR(64),
    subagent_judgment JSON,      -- SubAgent 上报内容
    mainagent_judgment JSON,     -- MainAgent 确认结果
    action_taken VARCHAR(32),    -- pass/degrade/block
    confirm_time DATETIME,
    INDEX idx_cockpit_time (cockpit_id, alert_time)
);

-- 审计日志表（14.1.3 #21: 所有车控指令/Agent确认/配置修改写入审计）
CREATE TABLE audit_logs (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    cockpit_id   VARCHAR(32) NOT NULL,
    user_id      VARCHAR(64) NOT NULL,
    action       VARCHAR(64) NOT NULL,    -- vehicle_cmd/config_change/agent_confirm/...
    detail       JSON,                    -- 操作详情
    ip_address   VARCHAR(45),
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_cockpit_time (cockpit_id, created_at),
    INDEX idx_user_time (user_id, created_at)
);

-- 用户反馈表（14.1.2 #10: 用户对 Agent 判断的反馈，用于优化 SubAgent prompt）
CREATE TABLE agent_feedback (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    cockpit_id   VARCHAR(32) NOT NULL,
    user_id      VARCHAR(64) NOT NULL,
    mainagent_log_id BIGINT,              -- 关联的 MainAgent 确认记录
    feedback     ENUM('positive', 'negative') NOT NULL,  -- 👍 / 👎
    comment      TEXT,                    -- 可选: 用户文字反馈
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_cockpit_time (cockpit_id, created_at),
    FOREIGN KEY (mainagent_log_id) REFERENCES mainagent_logs(id) ON DELETE SET NULL
);

-- LLM 成本追踪表（14.1.3 #19: 按座舱统计 LLM token 消耗与费用）
CREATE TABLE llm_cost_tracking (
    id           BIGINT AUTO_INCREMENT PRIMARY KEY,
    cockpit_id   VARCHAR(32) NOT NULL,
    request_type VARCHAR(32) NOT NULL,    -- chat/subagent/mainagent/asr/tts
    model_name   VARCHAR(64) NOT NULL,
    prompt_tokens INT DEFAULT 0,
    completion_tokens INT DEFAULT 0,
    cost_yuan    DECIMAL(10, 6) DEFAULT 0,  -- 单次调用费用（元）
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_cockpit_time (cockpit_id, created_at),
    INDEX idx_type_time (request_type, created_at)
);
```

> **SubAgent 异常模式记忆库**（14.1.2 #7）：不使用 MySQL 表，而是在 Milvus 中创建专用 collection `subagent_anomaly_memory`，存储历史判断正确的异常模式向量。字段：`cockpit_id` (TAG)、`anomaly_type` (TAG)、`description` (TEXT)、`judgment` (TEXT)、`embedding` (VECTOR)。下次巡检时先向量检索匹配，命中则跳过 LLM 调用。

## 八、技术选型与中间件对比

### 8.1 开发语言选型 — 三语言组合方案

> v2.1 引入 **TypeScript + Python + Go** 三语言架构，各司其职，兼顾 AI 生态、高并发与轻量化。

| 层 | 语言 | 框架 | 职责 | 选型理由 |
|----|------|------|------|---------|
| **前端 + BFF** | TypeScript | Next.js | UI 渲染、页面路由、BFF 数据聚合 | v2.0 已有，App Router 动态路由完美支持 `/cockpit/[cockpitId]` |
| **AI 智能层** | Python 3.11 | LangGraph + FastAPI | Agent 编排、RAG 检索、ASR/TTS、LLM 调用 | LangGraph/LangChain/FunASR/CosyVoice 生态锁死，无可替代 |
| **并发网关层** | Go 1.22 | Gin + gRPC | API 网关、WebSocket Hub、限流调度、实时推送 | goroutine 天然高并发、单二进制部署、极低内存占用 |

### 8.2 为什么选择 TS + Python + Go？

#### 8.2.1 候选语言评估

| 语言 | 并发模型 | 内存占用 | 编译/启动 | 学习曲线 | AI 生态 | Web 生态 | 适合本项目角色 |
|------|---------|---------|----------|---------|--------|---------|-------------|
| **TypeScript** | 事件循环 | 中 | 无需编译 | 低 | ❌ | ✅✅✅ | 前端 + BFF |
| **Python** | asyncio (GIL) | 高 | 解释执行 | 低 | ✅✅✅ | ✅✅ | AI 智能层 |
| **Go** | goroutine (CSP) | 极低 | 编译快 (秒级) | 中 | ❌ | ✅✅✅ | 并发网关层 |
| Rust | async + 零成本 | 最低 | 编译慢 (分钟级) | 极高 | ❌ | ✅✅ | ⚠️ 过度设计 |
| C++ | 线程池 | 最低 | 编译慢 | 极高 | ❌ | ✅ | ❌ 不适合 |

#### 8.2.2 为什么不是 Rust？

| 维度 | Go | Rust | 结论 |
|------|-----|------|------|
| 并发性能 | goroutine 10万级轻量 | async tokio 10万级 | 持平 |
| 内存占用 | ~20MB (网关服务) | ~5MB (网关服务) | Rust 胜，但 Demo 阶段差异不大 |
| 开发速度 | 快（语法简单、编译秒级） | 慢（生命周期、编译分钟级） | **Go 胜**，个人开发者关键 |
| 学习曲线 | 1-2 周上手 | 1-3 月上手 | **Go 胜** |
| 生态成熟度 | Web 框架极其成熟 | Axum/Actix 可用但社区小 | **Go 胜** |
| 部署友好 | 单二进制、交叉编译 | 单二进制 | 持平 |

**结论**：Go 在并发能力上与 Rust 接近，但在开发速度和学习曲线上远优于 Rust。对于个人开发者 Demo 项目，Go 是并发网关层的最优选择。Rust 的极致性能优势在 3 座舱 Demo 阶段无法体现，反而拖慢开发节奏。

#### 8.2.3 为什么不是 C++？

C++ 在本项目场景中完全没有优势：Web 服务生态不如 Go，AI 生态不如 Python，学习成本和开发效率远低于其他选项。仅在嵌入式车机端侧推理场景才有引入价值（v3.0+ 边缘部署时再评估）。

| **Python 不碰高并发** | Python 只接收 Go 转发的请求，处理 AI 逻辑后返回，不直接面对前端高并发 |

#### 8.2.7 Go ↔ Python 通信方案

```
方案: gRPC + Protobuf（高性能，强类型）

┌──────────┐     gRPC      ┌──────────┐
│  Go 网关  │ ──────────→  │ Python   │
│          │  agent.proto  │ AI 服务   │
│          │               │          │
│  路由完成  │  ←──────────  │  AI 结果  │
│  鉴权完成  │  response    │  返回     │
│  限流完成  │              │          │
└──────────┘               └──────────┘

优势:
• gRPC 二进制协议，比 JSON HTTP 快 5-10 倍
• Protobuf 强类型，Go 和 Python 自动生成客户端代码
• 流式 gRPC 支持 SSE 式流式对话（比 HTTP SSE 更高效）
• Go grpc-go 和 Python grpcio 都是成熟库

Demo 简化方案:
• Phase 1 先用 HTTP 反向代理（Go → Python FastAPI）
• Phase 2 逐步迁移到 gRPC（先迁移对话接口，再迁移其他）
```

| `WS /cockpit/{id}/ws/chat` | ✅ 连接管理 | ✅ Agent 流式 | Go 管理 WS，gRPC 调 Python |
| `WS /dataplatform/ws/realtime` | ✅ 实时推送 | | Go 直接推送指标变化 |

> **关键设计**：Go 网关直接处理所有非 AI 请求（健康检查、统计查询、中间件状态、设置 CRUD），只有需要 Agent/RAG/ASR/TTS 的请求才转发给 Python。这样 Python 服务只关注 AI 计算，不被简单 CRUD 请求打扰。

#### 8.2.9 三语言性能对比

| 指标 | 纯 Python (v2.0) | TS + Python + Go (v2.1) | 提升 |
|------|-----------------|------------------------|------|
| API 网关延迟 | ~5ms (FastAPI 路由) | ~0.5ms (Go Gin 路由) | 10x |
| WebSocket 并发上限 | ~500 连接 (asyncio) | ~10000 连接 (goroutine) | 20x |
| 网关内存占用 | ~200MB (FastAPI) | ~20MB (Go) | 10x |
| 简单 CRUD QPS | ~1000 (Python) | ~50000 (Go) | 50x |
| AI 请求 QPS | ~50 (LLM 瓶颈) | ~50 (LLM 瓶颈不变) | 持平 |
| 冷启动时间 | ~5s (Python import) | ~0.1s (Go 二进制) | 50x |

> **注意**：AI 请求的 QPS 瓶颈在 LLM API 调用，不在语言层。Go 网关的价值在于：非 AI 请求（统计/状态/CRUD）极快，WebSocket 连接管理极轻，Python 服务专注 AI 不被打扰。

### 8.3 中间件选型对比与推荐

#### Redis（缓存 + 限流 + Pub/Sub）

| 方案 | 优点 | 缺点 | 推荐? |
|------|------|------|------|
| **Redis Stack**（v2.0 已用） | RediSearch KNN 向量缓存、Pub/Sub 天然支持 SubAgent 上报 | 单机内存限制 | ✅ Demo 首选 |
| KeyDB | 多线程 Redis | 生态兼容性差 | ❌ |
| DragonflyDB | 高性能 Redis 替代 | 较新、文档少 | ❌ |
| Redis Cluster | 分布式 | Demo 过重 | ❌ |

**推荐**：继续用 Redis Stack。多座舱通过 DB 编号 + key 前缀隔离，Pub/Sub 做SubAgent→MainAgent 通信。

#### Milvus（向量检索）

| 方案 | 优点 | 缺点 | 推荐? |
|------|------|------|------|
| **Milvus 2.4**（v2.0 已用） | HNSW 索引、collection 天然隔离 | 资源占用大 | ✅ Demo 首选 |
| Qdrant | 轻量、Rust 写的 | 生态不如 Milvus | ⚠️ 备选 |
| Chroma | 极简、嵌入式 | 不适合多座舱 | ❌ |
| pgvector | 复用 PostgreSQL | 性能不如专用向量库 | ❌ |

**推荐**：继续用 Milvus。每个座舱用独立 collection 或共享 collection + `cockpit_id` 过滤。

#### RabbitMQ（任务队列）

| 方案 | 优点 | 缺点 | 推荐? |
|------|------|------|------|
| **RabbitMQ**（v2.0 已用） | vhost 天然隔离、管理界面好 | 吞吐不如 Kafka | ✅ Demo 首选 |
| Kafka | 高吞吐 | Demo 过重、运维复杂 | ❌ |
| Redis Streams | 无需额外中间件 | 不如专业 MQ 可靠 | ⚠️ 极简方案 |
| Celery 直接用 Redis | 减少中间件 | 功能受限 | ⚠️ 最简方案 |

**推荐**：继续用 RabbitMQ。Demo 简化为单 vhost + routing key 前缀隔离。

#### MySQL（持久化）

| 方案 | 优点 | 缺点 | 推荐? |
|------|------|------|------|
| **MySQL 8.0**（v2.0 已用） | 行级隔离简单（cockpit_id 列）、JSON 支持 | 单机限制 | ✅ Demo 首选 |
| PostgreSQL | 更强的 JSON/分区 | v2.0 用的是 MySQL | ⚠️ 需迁移 |
| SQLite | 零运维 | 不支持并发写 | ❌ 多座舱不适合 |

**推荐**：继续用 MySQL。所有表添加 `cockpit_id` 列做行级隔离。

#### Neo4j（知识图谱）

| 方案 | 优点 | 缺点 | 推荐? |
|------|------|------|------|
| **Neo4j 5.x**（v2.0 已用） | 节点属性隔离简单 | 资源占用 | ✅ Demo 首选 |
| ArangoDB | 多模型 | Cypher 生态更好 | ❌ |

**推荐**：继续用 Neo4j。节点添加 `cockpit_id` 属性做查询过滤。

### 8.4 新增技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 多租户上下文 | Python contextvars | 标准库、协程安全、零依赖 |
| SubAgent 调度 | asyncio 后台任务 | 不引入 Celery 复杂度，Demo 够用 |
| 实时数据推送 | WebSocket（已有） | 数据中台实时刷新用轮询即可，Demo 不需要 SSE |
| 图表库 | Recharts（已有） | 座舱对比/并发趋势图表 |
| 并发网关层 | Go 1.22 + Gin | goroutine 万级并发、单二进制部署、~20MB 内存 |
| Go ↔ Python 通信 | gRPC + Protobuf | 二进制协议高效、强类型、流式支持（Phase 2 迁移） |
| RBAC 权限 | Go JWT 中间件 + MySQL role 列 | 网关层统一鉴权、Python 信任 Go 传入的 user_context |
| 声纹识别 | Python CAM++ (3D-Speaker) | 复用 v2.0 已集成模型、按座舱隔离特征库 |
| WebSocket Hub | Go gorilla/websocket | 千级连接管理、每连接 ~4KB |

### 14.2 架构风险与最优方案

#### 风险 1：SubAgent LLM 调用成本

**问题**：3 座舱 × 每分钟巡检 × 每次调 LLM = 4320 次/天，LLM API 费用高昂。

**最优方案：三层降本策略**

```
┌──────────────────────────────────────────────────────────┐
│              SubAgent 降本策略（三层过滤）                   │
│                                                          │
│  Layer 1: 规则引擎预过滤（零 LLM 成本）                     │
│  ┌────────────────────────────────────────────────────┐  │
│  │  采集状态 → 规则判断（阈值/趋势）                      │  │
│  │  • 缓存命中率 > 80% → 正常，跳过 LLM               │  │
│  │  • P95 延迟 < 500ms → 正常，跳过 LLM               │  │
│  │  • 错误率 < 5% → 正常，跳过 LLM                    │  │
│  │  • 所有指标正常 → 不调 LLM（省 90% 调用）            │  │
│  │  → 只有指标越阈值才进入 Layer 2                     │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓ 越阈值                          │
│  Layer 2: 向量记忆库匹配（零 LLM 成本）                     │
│  ┌────────────────────────────────────────────────────┐  │
│  │  异常状态向量 → 检索历史异常记忆库                     │  │
│  │  • 相似度 > 0.95 → 直接复用历史判断结果               │  │
│  │  • 相似度 0.80~0.95 → 标记「疑似」进入 Layer 3       │  │
│  │  • 相似度 < 0.80 → 进入 Layer 3                    │  │
│  │  → 已知异常模式不重复调 LLM（再省 5%）               │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓ 未知异常                        │
│  Layer 3: LLM 深度判断（仅未知异常）                        │
│  ┌────────────────────────────────────────────────────┐  │
│  │  调用 LLM 分析异常 → 判断结果存入记忆库               │  │
│  │  → 只有真正未知的异常才消耗 LLM token                │  │
│  │  → 预计每天 < 50 次调用（vs 原始 4320 次）           │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  Demo 阶段额外措施:                                        │
│  • 巡检间隔从 30-60s 放宽到 5 分钟（正常状态）              │
│  • 异常苗头时自动加速到 30s                                │
│  • SubAgent 使用更便宜的模型（如 DeepSeek-V3 → Qwen-Turbo）│
│  • 预计日成本: < 0.5 元（vs 原始方案 ~15 元）              │
└──────────────────────────────────────────────────────────┘
```

#### 风险 2：MainAgent 确认延迟

**问题**：MainAgent 确认过程中用户请求被阻塞，体验差。

**最优方案：快慢双通道 + 异步流水线**

```
┌──────────────────────────────────────────────────────────┐
│           MainAgent 确认层 — 快慢双通道架构                 │
│                                                          │
│  用户请求到达                                             │
│       │                                                  │
│       ▼                                                  │
│  ┌────────────────────────────────┐                      │
│  │  快通道 (Fast Path) < 50ms     │                      │
│  │                                │                      │
│  │  1. 查 Redis 缓存的告警状态     │                      │
│  │     key: "alert:{cockpit_id}"  │                      │
│  │  2. 无告警 → 立即放行 ✅        │                      │
│  │  3. 有告警 → 检查告警状态:      │                      │
│  │     - status=confirmed_block   │                      │
│  │       → 立即拦截，返回降级提示  │                      │
│  │     - status=confirmed_pass    │                      │
│  │       → 立即放行 ✅             │                      │
│  │     - status=pending           │                      │
│  │       → 进入慢通道              │                      │
│  └────────────┬───────────────────┘                      │
│               │ pending                                  │
│               ▼                                          │
│  ┌────────────────────────────────┐                      │
│  │  慢通道 (Slow Path) < 3s       │                      │
│  │                                │                      │
│  │  1. 调用 LLM 二次确认           │                      │
│  │  2. 确认结果写入 Redis 缓存     │                      │
│  │     （后续相同告警走快通道）     │                      │
│  │  3. 用户请求等待结果:           │                      │
│  │     - 确认放行 → 放行 ✅        │                      │
│  │     - 确认拦截 → 返回降级提示   │                      │
│  └────────────────────────────────┘                      │
│                                                          │
│  关键设计:                                                 │
│  • 99% 请求走快通道（无告警时 < 50ms）                     │
│  • 慢通道仅在有 pending 告警时触发                         │
│  • 慢通道结果缓存 5 分钟，相同告警不重复调 LLM              │
│  • 慢通道超时（>3s）默认放行 + 后台继续确认                 │
│  • 确认结果异步通知前端（WebSocket / SSE）                 │
└──────────────────────────────────────────────────────────┘
```

#### 风险 3：多座舱资源竞争

**问题**：3 个座舱同时请求 LLM / 向量检索 / 车控，资源竞争导致性能下降。

**最优方案：分层限流 + 公平调度 + 优先级队列**

```
┌──────────────────────────────────────────────────────────┐
│              多座舱资源竞争解决方案                         │
│                                                          │
│  ┌─── Layer 1: 座舱级令牌桶限流 ──────────────────────┐  │
│  │                                                    │  │
│  │  每个座舱独立令牌桶:                                 │  │
│  │  • cockpit-01: 10 QPS (默认配额)                   │  │
│  │  • cockpit-02: 10 QPS                              │  │
│  │  • cockpit-03: 10 QPS                              │  │
│  │  • 全局上限: 30 QPS (保护 LLM API)                 │  │
│  │                                                    │  │
│  │  超出配额 → 排队等待（最多 5s）→ 超时返回 429       │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓                                │
│  ┌─── Layer 2: 优先级队列 (RabbitMQ) ────────────────┐  │
│  │                                                    │  │
│  │  消息优先级:                                        │  │
│  │  • P0: 车控指令（安全相关，最高优先）                │  │
│  │  • P1: 用户对话（实时性要求高）                     │  │
│  │  • P2: SubAgent 巡检（可延迟）                      │  │
│  │  • P3: 后台任务（记忆存储/统计聚合）                │  │
│  │                                                    │  │
│  │  公平调度: 同优先级按座舱轮询（防饥饿）              │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓                                │
│  ┌─── Layer 3: 资源熔断降级 ─────────────────────────┐  │
│  │                                                    │  │
│  │  LLM 调用熔断:                                      │  │
│  │  • 连续 5 次超时 → 熔断 30s → 降级为缓存/规则回复   │  │
│  │  • 熔断期间 SubAgent 停止 LLM 调用                  │  │
│  │                                                    │  │
│  │  向量检索熔断:                                      │  │
│  │  • Milvus 不可达 → 降级为 BM25 全文检索             │  │
│  │  • Neo4j 不可达 → 跳过图谱路径，仅用向量+BM25       │  │
│  │                                                    │  │
│  │  车控适配器熔断:                                    │  │
│  │  • HTTP 模式超时 → 3 次重试 → 降级为 Mock 模式     │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

#### 风险 4：数据隔离泄漏

**问题**：座舱 A 的用户可能看到座舱 B 的数据。

**最优方案：四层隔离保障**

```
┌──────────────────────────────────────────────────────────┐
│              数据隔离四层保障体系                          │
│                                                          │
│  ┌─── Layer 1: 中间件级隔离（物理/逻辑）──────────────┐  │
│  │                                                    │  │
│  │  Redis:     key 前缀 "cockpit-{id}:" 双重隔离      │  │
│  │  Milvus:    collection 按 cockpit_id 命名          │  │

│  │  RabbitMQ:  routing key "cockpit.{id}." 前缀       │  │
│  │  MySQL:     所有表 cockpit_id 列 + 复合索引         │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓                                │
│  ┌─── Layer 2: 应用层隔离（请求上下文）───────────────┐  │
│  │                                                    │  │
│  │  TenantContext (contextvars):                       │  │
│  │  • 每个 HTTP 请求设置 cockpit_id 到上下文            │  │
│  │  • 所有 Service/Repository 自动读取上下文 cockpit_id │  │
│  │  • 开发者无需手动传参，框架自动注入                  │  │
│  │                                                    │  │
│  │  Repository 层强制注入:                              │  │
│  │  • SELECT ... WHERE cockpit_id = %s  (MySQl)       │  │
│  │  • key = f"cockpit-{cid}:{key}"     (Redis)        │  │
│  │  • collection = f"cockpit_{cid}_memory" (Milvus)   │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓                                │
│  ┌─── Layer 3: JWT 鉴权隔离（身份绑定）────────────────┐  │
│  │                                                    │  │
│  │  JWT Token Payload:                                 │  │
│  │  {                                                  │  │
│  │    "user_id": "user_01",                            │  │
│  │    "cockpit_id": "cockpit-01",  ← 嵌入座舱 ID      │  │
│  │    "role": "user",                                  │  │
│  │    "exp": ...                                       │  │
│  │  }                                                  │  │
│  │                                                    │  │
│  │  API 中间件校验:                                    │  │
│  │  • token.cockpit_id == path.cockpit_id ?           │  │
│  │  • 不匹配 → 403 Forbidden                          │  │
│  │  • 管理员角色 (role=admin) 可跨座舱访问              │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↓                                │
│  ┌─── Layer 4: 自动化测试保障（持续验证）──────────────┐  │
│  │                                                    │  │
│  │  隔离测试用例:                                      │  │
│  │  • test_cockpit_isolation_redis:                   │  │

│  │    座舱1插入向量 → 座舱2 检索不应返回               │  │

│  │    座舱1插入记录 → 座舱2 查询不应返回               │  │

│  │    座舱1创建节点 → 座舱2 查询不应返回               │  │
│  │                                                    │  │
```

#### P1 — Agent 智能进化（v2.2+）

| 方向 | 说明 | 独特价值 |
|------|------|---------|
| **Agent 经验进化** | SubAgent 记忆库持续积累，越用越准 | 市面产品多为规则驱动，无自学习能力 |
| **Agent 主动建议** | 基于用户习惯主动推送服务（如定时预热车控） | 从被动响应 → 主动服务，差异化核心 |
| **Agent 协作广播** | 座舱间共享交通/天气/路况情报 | 多座舱协同是市面罕见能力 |
| **Agent 人格定制** | 每个座舱 Agent 独立性格、语气、回复风格 | 个性化体验，类似车载版「Siri 人设」 |
| **用户反馈闭环** | 👍👎 反馈 → SubAgent prompt 优化 | 形成 Agent 自我改进飞轮 |
| **置信度分级** | Agent 输出 confidence 分数，低置信度转人工 | 安全关键场景的可靠保障 |

#### P2 — 架构与部署扩展（v2.3+）

| 方向 | 说明 | 独特价值 |
|------|------|---------|
| **座舱横向扩展** | Kubernetes 部署，每座舱独立 Pod + 独立资源配额 | 支持从 3 → 100+ 座舱 |
| **跨座舱知识联邦** | 座舱间共享非敏感知识（如交通路况），敏感数据不出舱 | 联邦学习思想在座舱场景的应用 |
| **模型路由** | 不同座舱/不同任务使用不同 LLM 模型（车控用小模型快速响应，闲聊用大模型） | 成本与质量的最优平衡 |
| **边缘部署** | 座舱 Agent 部署到车载终端，断网可用 | 真实车控场景必需 |
| **移动端适配** | 座舱面板适配手机/平板，远程查看座舱状态 | 远程监控能力 |

### 开发环境 vs 生产环境端口流向

```
开发环境 (npm run dev + go run + python -m nexus.main):

  浏览器 ──→ Next.js Dev :3000 ──proxy──→ Go 网关 :8080 ──→ Python AI :8000
                                          ├──→ Redis / MySQL / Milvus ...
                                          └──→ WebSocket Hub :8080

  • Next.js dev server 的 rewrites 将 /api/* 代理到 Go 网关 :8080
  • WebSocket 连接直接指向 Go :8080
  • Python :8000 仅监听 127.0.0.1，不对外暴露

生产环境 (Docker Compose / Kubernetes):

  浏览器 ──→ Go 网关 :8080 ──→ Python AI :8000
                 ├──→ Redis / MySQL / Milvus ...
                 ├──→ WebSocket Hub :8080
                 └──→ Next.js 静态文件 (Go 内嵌或 Nginx 代理)

  • Go 网关作为唯一入口，serve 前端静态文件 + API 路由
  • Next.js 使用 next export 生成静态文件，Go 通过 http.FileServer 提供
  • Python :8000 仅在 Docker 内网可达，不绑外部端口
  • WebSocket 连接指向 Go :8080
```
