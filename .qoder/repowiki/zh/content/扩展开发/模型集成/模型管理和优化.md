## 简介
本文件面向NexusCockpit系统的“模型管理与优化”，围绕以下目标展开：
- 模型版本管理：元数据、依赖追踪、回滚策略
- 热更新机制：无缝切换、流量迁移、状态同步
- 降级策略：主备模型切换、性能降级、功能降级
- 监控与优化：指标采集、资源监控、自动扩缩容、成本优化
- 性能调优：缓存策略、批处理、GPU加速、量化压缩
- 故障排查与最佳实践

说明：仓库中未提供显式的“模型注册表/版本库”实现，但存在大量可支撑上述能力的模块（如RAG检索器/向量库/重排器工厂、技能编排、中间件缓存、可观测性指标等）。本文基于现有代码进行能力映射与实践建议。

## 核心组件
- 模型与状态
  - 模型定义与Schema：用于描述模型输入输出、参数、版本等元数据的载体
  - 运行态状态机：维护模型加载、就绪、热更、降级等生命周期状态
- 运行时编排
  - 意图路由：根据用户请求选择专家/技能/外部模型
  - 技能编排：组合ASR/TTS/RAG/车辆控制等能力
  - 会话与任务：会话存储、异步任务队列
- 外部模型与工具
  - 统一检索器：聚合向量库与图数据库检索
  - 重排器工厂：动态选择重排策略
  - 向量/图存储：Milvus/Zilliz、Neo4j/Aura等
- 中间件与核心
  - Redis缓存：热点结果、上下文缓存
  - 熔断器：保护下游不稳定服务
  - 限流与会话：保障稳定性
- 可观测性
  - 指标采集：延迟、吞吐、错误率、资源使用
  - 日志与链路追踪：便于定位问题

## 架构总览
下图展示一次典型对话的端到端流程，体现模型在其中的角色与交互点。

```mermaid
sequenceDiagram
participant C as "客户端"
participant API as "API路由(chat.py)"
participant INT as "意图路由(llm_router.py)"
participant RESP as "应答器(responder.py)"
participant ORCH as "技能编排(orchestrator.py)"
participant RAG as "统一检索(unified_retriever.py)"
participant VDB as "向量库(vector_store.py)"
participant GDB as "图数据库(graph_store.py)"
participant RR as "重排器(reranker_base.py)"
participant CACHE as "Redis缓存(redis_cache.py)"
participant OBS as "指标(observability)"

API->>INT : "解析意图"
INT-->>API : "返回目标专家/技能"

RESP->>ORCH : "执行编排"
ORCH->>CACHE : "查询缓存"
```

### 热更新机制（无缝切换、流量迁移、状态同步）
- 无缝切换
  - 新模型预热加载至独立实例或进程，待健康检查通过后替换活跃版本
  - 借助状态对象标记is_active与phase，确保原子切换
- 流量迁移
  - 通过网关或路由层按权重分流，逐步放量
  - 结合熔断器与限流，防止雪崩
- 状态同步
  - 会话上下文与缓存键包含模型版本标识，避免跨版本脏读
  - 使用Redis缓存隔离不同版本的热点数据

## 性能考虑
- 缓存优先：尽可能命中缓存，减少重复计算
- 批处理：合并相似请求，降低I/O与序列化开销
- 降级开关：在高峰期关闭非核心能力，保证核心体验
- 资源隔离：为不同模型/技能分配独立资源池，避免相互影响
- 容量规划：依据峰值QPS与延迟目标，预留冗余

## 故障排查指南
- 常见问题
  - 缓存不一致：检查缓存键是否包含版本与上下文指纹
  - 检索超时：检查向量库/图数据库连接池与慢查询

  - 熔断频繁触发：查看下游错误率与延迟分布
- 定位步骤
  - 查看指标与日志，定位瓶颈环节
  - 回放请求，复现问题
  - 临时降级或回滚，恢复服务
  - 修复后灰度发布，观察指标
