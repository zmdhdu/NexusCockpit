本仓库采用按子项目独立管理、双清单并存加锁文件的依赖管理模式，覆盖 Python、Go、Node.js 三种语言栈。

## 1. 使用的系统与工具
- Python 后端：pip 加 requirements.txt（生产）、pyproject.toml（开发元数据与可选依赖），通过 Makefile 统一安装；PyTorch 因体积大单独以 --index-url 指定源安装。
- Go 网关：Go Modules（go.mod 加 go.sum），无 vendor 目录，依赖直接从官方代理拉取。
- 前端 Next.js：npm（package.json 加 package-lock.json）。
- 基础设施编排：docker-compose.yml 集中声明所有中间件镜像版本（Milvus、Neo4j、Redis、RabbitMQ、MySQL、Prometheus、Grafana、Loki），作为运行时依赖的版本契约。

## 2. 关键文件与位置
- Python 开发元数据：backend_design/pyproject.toml，使用 >= 宽松约束，仅用于本地开发与可选依赖分组。
- Python 生产环境：backend_design/requirements.txt，全部使用 == 精确锁定，含 PyTorch 安装指引注释。
- Python 轻量环境：backend_design/requirements_no_torch.txt，屏蔽 torch/funasr/sentence-transformers 等重型依赖，便于快速启动。
- Go 模块：backend_design/nexus_gate/go.mod 与 go.sum，直接 require 第三方包，未启用 GOPROXY/GOPRIVATE 自定义代理。
- Node.js：frontend_design/package.json 与 package-lock.json，生产与 devDependencies 分离。
- 基础设施镜像：docker-compose.yml，所有中间件服务以固定 image tag 声明。
- 安装入口：Makefile，封装 pip/npm/docker compose 命令，提供 install / install-gpu / install-frontend / install-all。

## 3. 架构与约定
- 双清单分层：pyproject.toml 面向开发者（宽松范围加 optional-dependencies），requirements.txt 面向构建与部署（全量 == 锁定），二者共同构成可复现的依赖契约。
- PyTorch 解耦：通过 requirements.txt 顶部注释与 Makefile 的 install 与 install-gpu 两条路径分别指向 CPU/GPU wheel 源，避免在 CI 或无 GPU 环境中误装 CUDA 依赖。
- 按需裁剪：requirements_no_torch.txt 将 ASR/TTS/NLP 相关依赖注释掉，使纯 API 场景可在内存受限环境快速拉起。
- Go 不 vendoring：nexus_gate 直接使用 go.sum 校验，依赖由 go mod download 拉取；未见 GOPROXY、GOPRIVATE 等私有代理配置。
- 前端 lockfile 入仓：package-lock.json 随代码提交，保证 npm ci 可完全复现。
- Docker Compose 即依赖清单：所有外部服务（向量库、图数据库、缓存、消息队列、数据库、观测栈）均以固定镜像版本声明，配合 healthcheck 保障启动顺序。

## 4. 开发者应遵循的规则
- 新增 Python 依赖时：同时在 pyproject.toml（宽松范围）和 requirements.txt（精确版本）中声明，并在 requirements_no_torch.txt 中评估是否需要屏蔽。
- 不要手动编辑 requirements.txt 以外的 pip 配置文件：当前仓库未使用 pip.conf 或 .pip/pip.conf 或 poetry.lock，所有源与索引通过 Makefile 硬编码。
- Go 依赖更新：在 backend_design/nexus_gate/ 下执行 go get -u 后提交 go.mod 与 go.sum，无需 vendor。
- 前端依赖更新：在 frontend_design/ 下执行 npm i 并检查 package-lock.json 变更后再提交。
- 中间件升级：修改 docker-compose.yml 中的镜像 tag，并确保对应 healthcheck 命令仍有效。
- CI/CD 尚未集成依赖扫描：目前未见 .github/workflows/ci.yml 中对依赖漏洞扫描或自动更新的步骤，建议在流水线中加入 pip-audit、npm audit、govulncheck。
