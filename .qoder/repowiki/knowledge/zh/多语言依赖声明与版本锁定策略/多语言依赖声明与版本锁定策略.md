本仓库为全栈工程，包含 Python 后端、Go 网关与 Next.js 前端三个独立子项目，每个子项目各自维护其依赖清单，未使用统一的顶层依赖管理工具（如 pip-tools/poetry lock、go mod tidy -u、npm ci 等），也未见 vendor 目录或私有源配置。

1. Python 后端（backend_design）
   - 主清单：pyproject.toml，使用 PEP 621 风格声明 dependencies 与 optional-dependencies.dev，要求 Python >=3.10。
   - 兼容清单：requirements.txt 与 requirements_no_torch.txt 作为历史/轻量安装入口，但当前构建系统由 pyproject.toml 的 [build-system] 指定 setuptools 驱动。
   - 版本约束：全部采用 >=X.Y.Z 宽松下限，无固定上限，亦无 poetry.lock / pipenv.lock 等锁文件；开发依赖通过 dev extra 提供 pytest/ruff/mypy 等工具链。
   - 包发现：[tool.setuptools.packages.find] where = ["."] include = ["nexus*"] 仅打包 nexus* 命名空间。

2. Go 网关（backend_design/nexus_gate）
   - 模块清单：go.mod 声明 module nexus_gate、Go 1.22，直接依赖 gin、golang-jwt/v5、gorilla/websocket、prometheus/client_golang。
   - 版本锁定：存在 go.sum（在 go.mod 同目录下）用于校验间接依赖，但未启用 Go workspace 或 vendor 模式。
   - 私有源：未见 GOPRIVATE、proxy.golang.org 替换或自定义 GOPROXY 配置。

3. Next.js 前端（frontend_design）
   - 清单：package.json 以 dependencies + devDependencies 双区组织，使用 npm 语义化版本范围（^、~）。
   - 锁文件：仓库中未发现 package-lock.json 或 yarn.lock，无法保证跨环境确定性安装。
   - 脚本：提供 dev/build/start/lint/type-check 标准 npm scripts，未集成自动升级或安全扫描命令。

4. 编排与 CI 中的依赖安装
   - 根级 docker-compose.yml 负责拉起各服务镜像，但镜像构建阶段如何拉取依赖取决于各子项目的 Dockerfile（未在本文读取范围内确认是否使用 --frozen-lockfile 等策略）。
   - .github/workflows/ci.yml 作为统一 CI 入口，若其中调用 pip install/go build/npm ci，则构成实际依赖解析策略；当前未读取该文件内容，故不在此断言。

开发者应遵循的规则与建议：
- 新增依赖一律写入对应子项目的清单文件（pyproject.toml / go.mod / package.json），禁止在代码中硬编码版本号。
- 引入锁文件：Python 侧建议改用 poetry.lock 或 pip-tools 生成 requirements.txt 并配合 pip install -r；Go 侧保持 go.sum 提交；前端侧提交 package-lock.json 并使用 npm ci。
- 将版本范围收紧到最小必要区间（如 >=X,<Y），避免上游破坏性更新影响构建。
- 如需私有包源，应在各语言生态中分别配置（PyPI mirror、GOPROXY、npm registry），并在 CI 中以环境变量注入。
