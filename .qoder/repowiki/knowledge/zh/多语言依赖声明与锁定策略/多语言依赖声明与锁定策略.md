本仓库采用**多包管理器并存**的依赖管理方式，按语言域分别声明、锁定与安装：

1. **Python（后端 AI 服务）**
   - 双清单共存：`backend_design/pyproject.toml` 使用 `>=` 宽松范围声明，面向可复现构建；`backend_design/requirements.txt` 使用 `==` 精确版本，作为实际运行快照。
   - PyTorch/torchaudio 因体积大且分 CPU/GPU 索引，不在 requirements 中固定，改为通过注释 + 文档指引手动以 `--index-url` 指定源安装。
   - 未启用 pip 私有镜像或 `pip.conf`，仅通过命令行参数切换 PyTorch 官方索引。
   - 无 `requirements_no_torch.txt` 之外的额外约束文件，开发依赖同时混入 `requirements.txt`，未见独立 `dev-requirements.txt`。

2. **Go（NexusGate 网关）**
   - 使用标准 Go Modules：`backend_design/nexus_gate/go.mod` 声明模块名与直引依赖，`go.sum` 由工具生成（仓库中未提交 go.sum，需本地 `go mod tidy` 生成）。
   - 未发现 `GOPRIVATE`、`GONOSUMCHECK`、`proxy` 等代理/私有仓库配置，默认走 golang.org 公共索引。
   - 未使用 vendor 目录，依赖随构建时拉取。

3. **Node.js / TypeScript（Next.js 前端）**
   - 使用 npm v3 lockfile：`frontend_design/package.json` + `frontend_design/package-lock.json` 锁定所有子依赖及 sha512 integrity。
   - 从 lockfile 的 `resolved` 字段可见默认使用 `registry.npmmirror.com`（淘宝镜像），说明开发者环境已全局配置 npm 镜像。
   - 未使用 pnpm/yarn，也未见 `.npmrc` 或 `pnpm-workspace.yaml`。

4. **容器化与 CI 集成**
   - 顶层 `docker-compose.yml` 是统一编排入口，各组件在 Dockerfile 中各自执行对应语言的依赖安装命令，未引入统一的依赖缓存层。
   - GitHub Actions (`/.github/workflows/ci.yml`) 负责触发构建，但具体依赖安装步骤由各 Dockerfile 内定义。

**约定与建议**
- Python 侧建议将 `requirements.txt` 视为“冻结快照”，变更需同步更新 `pyproject.toml` 的版本范围并重新生成锁文件。
- Go 侧如需接入企业私有 GOPROXY，应在 `go.mod` 同级添加 `go.work` 或在 CI 环境变量中注入 `GOPROXY`、`GOPRIVATE`。
- Node 侧若团队共享镜像，可在仓库根增加 `.npmrc` 统一 registry 配置，避免依赖个人全局设置。
